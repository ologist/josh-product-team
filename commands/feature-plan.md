Josh의 제품팀 기능 기획 워크플로우를 실행하라.

입력: $ARGUMENTS (기능명, 요구사항 설명, 또는 관련 컨텍스트)

다음 4단계를 순서대로 완전히 수행하라.

---

## STEP 1: PM — 문제 정의 및 PRD 작성

🗂️ **PM 관점**으로 다음을 수행하라.

먼저 이 기능이 해결하는 핵심 문제를 정의하라:
- 사용자가 겪는 실제 문제는 무엇인가?
- 이 기능이 없으면 어떤 일이 발생하는가?
- 비즈니스 목표와의 연결고리는?

그 다음 PRD를 작성하라:

```
[🗂️ PRD: {기능명}]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

## 배경 & 목표
- 해결하는 문제: ...
- 비즈니스 목표: ...
- 성공 기준 (측정 가능한 지표): ...

## 사용자 & 페르소나
- 주요 사용자: ...
- 사용 맥락: ...

## 유저 스토리
- As a [...], I want to [...] so that [...]
- ...

## 기능 요구사항

### Must Have (MVP)
- [ ] ...

### Should Have
- [ ] ...

### Won't Have (이번 버전 제외)
- ...

## 엣지 케이스
- ...

## 성공 지표
- ...

## 범위 외
- ...
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

---

## STEP 2: 백엔드 엔지니어 — API & 데이터 설계

⚙️ **백엔드 엔지니어 관점**으로 PRD를 받아 기술 설계를 수행하라.

- 필요한 API 엔드포인트 설계 (Method, Path, Request/Response)
- DB 스키마 또는 데이터 모델 설계
- 인증/인가 요구사항
- 성능·보안 고려사항
- 예상 구현 공수 및 기술적 리스크

```
[⚙️ 백엔드 기술 스펙: {기능명}]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

## API 엔드포인트
| Method | Path | 설명 | Auth |
|--------|------|------|------|
| ... | ... | ... | ... |

## 핵심 API 상세
[주요 엔드포인트 Request/Response 스펙]

## 데이터 모델
[필요한 테이블 또는 컬렉션 구조]

## 기술적 고려사항
- 인증: ...
- 성능: ...
- 보안: ...

## 구현 예상 공수
- ...

## 백엔드 → PM 피드백
[PRD에서 불명확하거나 조정이 필요한 부분]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

---

## STEP 3: 프론트엔드 엔지니어 — UI/UX 설계

🎨 **프론트엔드 엔지니어 관점**으로 PRD와 백엔드 스펙을 받아 UI 설계를 수행하라.

- 컴포넌트 계층 구조
- 사용자 흐름 및 상태 처리 (로딩/에러/빈 상태)
- 상태 관리 전략
- 반응형 고려사항
- 구현 예상 공수 및 기술적 리스크

```
[🎨 프론트엔드 기술 스펙: {기능명}]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

## 컴포넌트 구조
[페이지/컴포넌트 계층]

## 사용자 흐름
[주요 인터랙션 시나리오]

## 상태 관리
- 서버 상태: ...
- 클라이언트 상태: ...

## UI 상태 처리
- 로딩: ...
- 에러: ...
- 빈 상태: ...

## 구현 예상 공수
- ...

## 프론트엔드 → PM/백엔드 피드백
[PRD나 API 설계에서 조정이 필요한 부분]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```
