Josh의 제품팀 스프린트 계획 워크플로우를 실행하라.

입력: $ARGUMENTS (백로그 항목 목록, 기능 설명, 또는 현재 스프린트 상황)

다음 3단계를 순서대로 수행하라.

---

## STEP 1: PM — 우선순위 결정

🗂️ **PM 관점**으로 제공된 항목들의 우선순위를 결정하라.

RICE 스코어를 사용하여 각 항목을 평가하라:
- **Reach**: 영향받는 사용자 수 (1-100)
- **Impact**: 목표 달성 기여도 (0.25 / 0.5 / 1 / 2 / 3)
- **Confidence**: 예측 신뢰도 (50% / 80% / 100%)
- **Effort**: 예상 개발 공수 (person-weeks, 다음 단계에서 엔지니어가 보정)

```
[🗂️ PM — 우선순위 결정]

| # | 기능/태스크 | Reach | Impact | Conf | Effort(추정) | RICE | 비고 |
|---|-----------|-------|--------|------|-------------|------|------|
| 1 | ... | ... | ... | ... | ... | ... | ... |

이번 스프린트 Must:
- ...

다음 스프린트로 이동:
- ...

제외 (이유 포함):
- ...
```

---

## STEP 2: 엔지니어링 팀 — 공수 산정

백엔드와 프론트엔드 엔지니어가 각 항목의 실제 공수를 산정하라.

### ⚙️ 백엔드 공수 산정
각 항목에 대해:
- 구현 복잡도 평가
- 실제 공수 (시간 또는 일 단위)
- 기술적 리스크 및 선행 작업

### 🎨 프론트엔드 공수 산정
각 항목에 대해:
- 구현 복잡도 평가
- 실제 공수 (시간 또는 일 단위)
- 디자인/백엔드 의존성

```
[⚙️🎨 공수 산정 결과]

| # | 기능/태스크 | 백엔드 | 프론트엔드 | 합계 | 의존성 | 리스크 |
|---|-----------|--------|----------|------|--------|--------|
| 1 | ... | ... | ... | ... | ... | ... |

총 예상 공수: 백엔드 ... / 프론트엔드 ...
스프린트 용량 대비: [적정 / 과부하 — 조정 필요]

기술적 리스크:
- ...
```

---

## STEP 3: 최종 스프린트 백로그

3인의 인풋을 통합하여 최종 스프린트 백로그를 생성하라.

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
📋 스프린트 백로그
기간: {스프린트 기간}
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

■ 이번 스프린트 목표
  ...

■ 확정 태스크 (우선순위 순)

  [백엔드]
  [ ] #{번호} {태스크명} — {공수}
  ...

  [프론트엔드]
  [ ] #{번호} {태스크명} — {공수}
  ...

  [공통/기타]
  [ ] ...

■ 스프린트 용량
  백엔드: {총 공수} / 용량 {가용 공수}
  프론트엔드: {총 공수} / 용량 {가용 공수}

■ 다음 스프린트 예비 목록
  - ...

■ 블로커 & 의존성
  - ...
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```
